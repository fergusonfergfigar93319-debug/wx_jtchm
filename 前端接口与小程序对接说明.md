# 今天吃什么小程序 — 前端接口需求与运行说明（前后端对接）

> **口径说明**：本文档以仓库内 `utils/api.js` 的实际请求为准。同目录下 `给后端开发.md` 为历史/扩展说明，若与本文件冲突，**以本文件与 `api.js` 为准**。

---

## 一、环境与统一约定

### 1.1 Base URL

- 配置位置：`app.js` → `globalData.apiBaseUrl`（默认 `http://127.0.0.1:8000/api/v1`）。
- 所有接口路径均为该前缀下的相对路径。

### 1.2 鉴权

- 需登录接口：请求头 `Authorization: Bearer {access_token}`。
- 登录成功后：`access_token`、`refresh_token` 写入 `wx.setStorageSync`（键名 `access_token`、`refresh_token`），并在 `app.globalData` 中同步；详见 `utils/auth.js`。
- 例外：`POST /users/login/` 等 `needAuth = false` 的接口不传 token。
- **401**：前端会清空本地会话并调用 `wxLogin()` 重新登录（见 `utils/api.js` 的 `request`）。

### 1.3 响应格式

- 成功：`{ code: 200, data?: any, message?: string, msg?: string }`。
- 前端逻辑：`code === 200` 时，若存在 `data` 则 **resolve `data`**，否则 resolve 整个 `res.data`。
- 业务失败：`code !== 200` 且非 401 → `reject(Error(message || msg || '请求失败'))`。
- **HTTP 非 200**：当前 `utils/api.js` 的 `request` **仅将 `statusCode === 200` 视为成功**并解析 `{ code, data }`；若后端对创建类接口返回 **201 Created**、删除成功返回 **204 No Content**，需在 `api.js` 中把成功条件放宽为 2xx（并兼容 204 无响应体），否则会误报「请求失败: 201/204」。

### 1.4 真实后端联调

- 已移除本地 Mock：`utils/mock.js` 不再存在，**所有接口均走真实网络**。
- 请在微信公众平台配置 **request 合法域名**（及 uploadFile 等所需域名），开发阶段可勾选「不校验合法域名」。

### 1.5 文件上传

- 使用 `wx.uploadFile`，响应体为 **字符串**，需 `JSON.parse`。
- 成功条件：`parsed.code === 200`，数据取 `parsed.data` 或整对象。

## 二、接口清单（与 `utils/api.js` 一致）

下表为前端**会发起**的请求；**请求体/Query** 根据页面调用整理，便于后端实现与联调。

### 2.1 用户与档案

| 方法 | 路径 | 说明 | 请求参数 / Body | 调用场景 |
|------|------|------|-----------------|----------|
| POST | `/users/login/` | 微信 `code` 换令牌 | `{ code }` | 返回 `data.access_token`、`data.refresh_token`、`is_new_user`、`user_id`；`login` 页、`app.wxLogin` |
| POST | `/users/logout/` | 退出（拉黑 refresh） | `{ refresh_token }` | Header：`Bearer access_token`；成功后再 `clearStorage` 并回登录页；`profile` 设置 |
| GET | `/diet/profile/` | 当前用户档案 | 无 | 全局、`profile-edit`、各页刷新 |
| PATCH | `/diet/profile/` | 更新档案 | 见下「档案 PATCH 字段」 | `profile-edit`、`profile` 部分快捷修改 |
| POST | `/diet/profile/` | **上传头像**（multipart） | 表单字段名 **`avatar`**，文件路径为本地临时路径 | `profile-edit`（`wx.uploadFile`） |
| GET | `/user/{userId}/profile/` | 他人公开主页 | 无 | `user-profile` |
| POST | `/user/{userId}/follow/` | 关注 | 无 body | `user-profile` |
| DELETE | `/user/{userId}/follow/` | 取消关注 | 无 body | `user-profile` |
| GET | `/user/{userId}/posts/` | TA 的动态列表 | Query：`page`, `page_size` | `user-profile` |

**档案 PATCH 常用字段**（`profile-edit` 提交示例）：`nickname`, `signature`, `height`, `weight`, `target_weight`, `daily_kcal_limit`, `goal_type`, 可选 `avatar`（URL 字符串，与上传结果衔接）。

**登录响应**（`app.js` 与 `api.login` 共用逻辑）：`data.token` 必填；`data.is_new_user` 为 true 时跳转档案编辑页。

### 2.2 智能推荐（搜餐）

| 方法 | 路径 | 说明 | Body（POST） |
|------|------|------|----------------|
| POST | `/diet/search/` | 菜谱/商家推荐 | 见下「searchRecommend 结构」 |

**`searchRecommend` 典型结构**（不同页面字段多少不一，后端宜做宽松解析）：

- **通用**：`mode`: `'cook'` | `'restaurant'`
- **位置**：`lat`, `lng`（可选，浮点数）
- **「自己做」`cook`**（`packageCook/cook/cook.js`）：
  - `use_fridge`: boolean
  - `filters`: `{ tags[], sort_by, keyword, cooking_time, difficulty, calorie_min, calorie_max, available_tools[], meal_prep }`
- **「点外卖」`restaurant`**（`packageRestaurant/restaurant/restaurant.js`）：
  - `lat`, `lng`（必填，否则前端提示先定位）
  - `filters`: `{ sort_by, keyword, health_status, nutrition_type }`
  - `page`, `page_size`（分页）
- **「帮我选」`smart-choice`**（`packageCook/smart-choice/smart-choice.js`）：
  - `meal_type`, `remaining_calories`, `macros`（`carbg`/`proteing`/`fatg` 等）, `filter_mode`, `meal_prep`, `count`
  - 匹配模式：`use_fridge`, `fridge_ingredients[]`（食材名字符串）
  - 应季联动：`seasonal_ingredient`, `from_seasonal_page`, `filters.keyword`
- **菜谱详情相关推荐**：`mode: 'cook'`, `filters: { tags, exclude_id }`

**响应**：前端读取 `recommendations` 或 `items` 数组；元素含 `type`（`recipe`/`restaurant`）、`id`、`name`、`calories`、`image`、`ingredients`、`tags` 等（与页面展示约定一致即可）。

## 三、小程序运行逻辑

### 3.1 启动流程（`app.js`）

1. `onLaunch`：检测运行平台、`liteMode`。
2. 约 100ms 后：`checkLogin()`：有 token → `getUserInfo()`；无 token → `wx.reLaunch('/pages/login/login')`。
3. 再延迟：`getUserLocation()`（`gcj02`），写入 `globalData.location`，失败静默。

### 3.2 登录页

- 用户触发微信登录后：`api.login(code)` → 存 token → 进入主页或新用户去 `profile-edit`。

### 3.3 主包 Tab（自定义 `custom-tab-bar`）

| Tab | 路径 | 作用 |
|-----|------|------|
| 首页 | `pages/index/index` | 今日摘要、快捷入口、推荐卡片 |
| 冰箱 | `pages/fridge/fridge` | 食材列表、识别入库、临期清理联动 |
| 健康管理 | `pages/report/report` | 日历、日报/周报、图表、饮水（本地 ml）、编辑摄入 |
| 我的 | `pages/profile/profile` | 档案、收藏入口、成就、设置类操作 |

### 3.4 分包与主要页面（`app.json`）

- **packageCook**：自己做、菜谱详情、帮我选、应季、食材识别、拍图识热量…
- **packageRestaurant**：外卖列表、商家详情、拼单（`group-order`）等
- **packageHealth**：摄入编辑、饮水、补救、triage、碳足迹、地图跑步…
- **packageCommunity**：动态流、发帖、帖子详情、用户主页、通知
- **packageUser**：档案编辑、收藏、购物清单、挑战、成就
- **packageAI**：AI 营养师、AI 对话

### 3.5 数据流小结

- **鉴权**：绝大多数 `api.request` 自动带 Bearer token。
- **位置**：外卖列表、部分推荐依赖 `globalData.location` 或页面内 `wx.getLocation`。
- **收藏**：部分场景**先写本地 storage** 再调后端，后端失败时仍保留本地（如菜谱详情收藏）。

## 五、联调检查清单

1. 所有接口返回 **`code: 200`** 且业务数据放在 **`data`**（与现有前端解析一致）。
2. **401** 统一处理 token 失效。
3. **`PATCH/DELETE`** 摄入记录路径为 **`/diet/log/{id}/`**。
4. 头像上传：**`wx.uploadFile`** 到 **`/diet/profile/`**，文件字段名 **`avatar`**。
5. **`POST /diet/search/`** 同时服务「自己做列表」「外卖列表」「帮我选」，通过 **`mode`** 与 **`filters`** 区分。
6. 社区发帖含 **图片本地路径** 时，后端若需 URL，需约定先上传 OSS 再提交 `images`（当前前端可能直接传临时路径，需产品/后端定稿）。
