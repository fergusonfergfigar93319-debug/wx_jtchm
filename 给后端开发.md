## 项目后端接口依赖总览

**后端 Base URL（统一前缀）**

- `http://127.0.0.1:8000/api/v1`
- 真实请求地址 = `apiBaseUrl` + 下文所有相对路径
- 统一响应约定（除特别说明外）：
  - 成功：`{ code: 200, data?: any, message?/msg?: string }`
  - 前端取值逻辑：优先取 `data` 字段，否则使用整个响应对象
  - 鉴权失败：`code === 401` 时前端会清 token 并重新触发登录

**统一请求封装（前端实现概要）**

- 文件：`utils/api.js`
- 核心方法：`request(url, method = 'GET', data = {}, needAuth = true)`
- 鉴权：`needAuth === true` 时会在 header 中附加 `Authorization: Bearer {token}`
- **请求**：`utils/api.js` 中 `request()` **始终发起真实网络请求**；已移除历史 Mock（无 `utils/mock.js`、无 `useMockData` 开关）。

    - `api.getUserProfile()` 中作为旧接口兜底
- **请求头**
  - `Authorization: Bearer {token}`
- **请求参数**：无
- **返回**

### 1.3 更新用户档案

### 1.4 上传头像

- **方法**：`POST /diet/profile/`（与档案更新同路径，`wx.uploadFile` 上传文件）
- **函数**：`api.uploadAvatar(filePath)`
- **表单**
  - 文件字段名：**必须**为 `avatar`（与小程序 `wx.uploadFile` 的 `name: 'avatar'` 一致）
  - 额外表单字段：无（与《前端接口与小程序对接说明》一致）
  - 请求体类型：`multipart/form-data`（**不是** JSON）
- **请求头**
  - `Authorization: Bearer {token}`（如果有 token）
- **返回（必须满足，否则小程序端无法展示新头像）**
  - HTTP 状态码：**2xx**（建议 200）
  - 响应体为 **JSON 字符串**（`wx.uploadFile` 的 `res.data` 是字符串，需可 `JSON.parse`）
  - 业务包装：**必须**包含 `code: 200`（与项目其余接口一致；若只返回 `{ success: true }` 而无 `code`，前端会判定失败）
  - **`data` 中必须能拿到头像地址**，至少满足其一（前端会递归解析 `data` / `profile` / `user` 等嵌套）：
    - 推荐：在 **`data` 内返回与 `GET /diet/profile/` 相同的用户档案对象**，且其中包含 **`avatar` 或 `avatar_url`**（值为**完整可访问 URL**，建议 `https://`）
    - 或：`data.avatar`、`data.avatar_url`、`data.file_url` 等字符串字段
  - 若 `POST` 响应里暂时没有头像 URL：**必须**保证紧接着的 **`GET /diet/profile/`** 能返回已更新后的 `avatar` / `avatar_url`（前端上传成功后会自动再 GET 一次拉最新档案）

**后端实现要点（给 Django / DRF 等）：**

1. **路由**：`POST /diet/profile/` 与 `PATCH /diet/profile/` 需区分：PATCH 收 JSON 更新字段；**POST 收 `multipart/form-data`，并从 `FILES['avatar']`（或等价）读取文件**。若当前仅注册了 PATCH 而未实现 POST 文件上传，会表现为 **405 Method Not Allowed** 或 **404**。
2. **持久化**：保存文件到对象存储或 `MEDIA`，并把**可对外访问的绝对 URL**（或经网关后的 HTTPS 路径）写回用户表的 `avatar` / `avatar_url` 字段。
3. **返回体示例（最小可用）**：
   ```json
   {
     "code": 200,
     "data": {
       "avatar": "https://your-domain.com/media/avatars/xxx.jpg",
       "nickname": "…",
       "height": 170,
       "weight": 60
     }
   }
   ```
4. **禁止**：仅返回 `{ "message": "ok" }` 而无 `code: 200`、无 `data`、无头像字段；或 `avatar` 为相对路径但小程序未配置 **downloadFile 合法域名**（若 `<image>` 使用网络图，域名需在小程序后台配置）。

**联调时如何区分前端 / 后端问题：**

- 在微信开发者工具 **Network** 中查看：
  - **`POST .../diet/profile/`**：若 HTTP 非 2xx、或响应体无法解析出 `code: 200`、或 `data` 中始终没有头像 URL → **后端响应不符合约定**。
  - **`GET .../diet/profile/`**：若 POST 显示成功但 GET 里 `avatar` 仍是旧值或空 → **后端未写入数据库或缓存未失效**。
  - 若两项均正常，仍不显示 → 再查 **图片 URL 是否 HTTPS**、是否被小程序 **downloadFile / 业务域名** 拦截（属配置问题）。

#### 6.2.1 后端待办（收藏列表 500 时必查）

若小程序或联调日志出现 **`GET /api/v1/diet/favorites/` → 500 Internal Server Error**，请在后端处理：

1. **在运行 `runserver` 的终端查看完整 traceback**，定位抛错行（常见：模型/外键不存在、未迁移、序列化字段为 `None`、对 `Preference`/`Favorite` 查询条件写错、`annotate`/`select_related` 引用了不存在的关联）。
2. **保证已登录用户**：视图应使用 `request.user`，且与 `POST /diet/preference/` 写入的收藏数据为同一套存储；空收藏时返回 `code:200` + `[]` 或 `{ list: [] }`，不要抛未捕获异常。
3. **自测**：`curl` 或 Apifox 带 `Authorization: Bearer <有效 token>` 请求 `GET .../diet/favorites/`，确认 HTTP 200 且 body 符合统一 `code/data` 约定。
4. 修复并部署后，前端会优先用接口数据；接口仍失败时个人页/收藏页会暂时用本机 `storage` 里的 `favorites` 降级展示，**与后端是否修好无关**，不能替代服务端修复。

## 11. 碳足迹

> 目标：**仅保留“绿色足迹”页（截图）所需能力**，其它扩展能力（明细可解释、环保建议、碳成就、纠错反馈等）全部先不做，以保证后端实现最小闭环、前端请求最少。
>
> 说明：本章节为**权威口径**。若后端已有旧版接口（`/carbon/footprint/*`），请参考 11.x 的“兼容映射”进行对齐（建议保留旧路径但内部转调）。

### 11.1 数据口径（最小可用）

- **总量单位**：`kg CO2e`（千克二氧化碳当量）
- **来源拆分（建议）**：
  - `food_kg`：饮食（来自饮食记录/菜谱/餐厅等）
  - `travel_kg`：出行（可选，若无数据则返回 `0`）
  - `package_kg`：包装（可选，若无数据则返回 `0`）
  - `sport_offset_kg`：运动抵消（负数，展示用，来自运动记录）
- **等级（强烈建议）**：
  - `level: 'low' | 'medium' | 'high'`
  - 由后端按阈值映射，阈值可配置/版本化；前端仅展示，不参与判断

### 11.2 获取“绿色足迹”页数据（今日/指定日期）

- **方法**：`GET /carbon/summary/`
- **函数**：建议新增 `api.getCarbonSummary(date?)`
- **鉴权**：需要
- **Query**
  - `date?: string`（`YYYY-MM-DD`，不传则取今日）
- **用途**：覆盖截图中的：
  - 今日碳足迹总量 + 等级
  - 食物/运输/包装拆分 + 运动抵消
  - 本周平均/本周合计
  - 今日运动卡片（距离/时长/消耗/碳抵消）
  - 运动轨迹（可选：若无轨迹则前端隐藏地图块）

`CarbonSummary` 建议结构：

```json
{
  "date": "2026-03-18",
  "total_kg": 3.42,
  "level": "low",
  "breakdown": {
    "food_kg": 2.80,
    "travel_kg": 0.40,
    "package_kg": 0.22,
    "sport_offset_kg": -0.60
  },
  "week": {
    "avg_kg_per_day": 3.10,
    "total_kg": 21.70
  },
  "today_workout": {
    "distance_km": 8.5,
    "duration_sec": 3600,
    "calories_kcal": 420,
    "offset_kg": -0.60,
    "track": {
      "provider": "tencent",
      "center": { "lat": 30.000001, "lng": 120.000001 },
      "polyline": [
        { "lat": 30.000001, "lng": 120.000001 },
        { "lat": 30.000500, "lng": 120.000900 }
      ]
    }
  }
}
```

字段说明（精简版）：

- `breakdown.*_kg`：均可为 `0`；`sport_offset_kg`/`today_workout.offset_kg` 为负数
- `week`：若后端不做周统计，可不返回该字段（前端隐藏“本周平均”卡片）
- `today_workout`：无运动时可为 `null` 或不返回（前端隐藏“今日运动/轨迹”块）
- `today_workout.track`：无轨迹（仅有运动摘要）时可不返回（前端隐藏地图，仅展示运动卡）

### 11.3 计算与降级（必须满足截图体验）

- **必须可返回**：`total_kg`、`level`、`breakdown`
- **可选返回**：`week`、`today_workout`
- **空数据降级**：当天没有任何记录时，返回 `total_kg = 0`、`level = 'low'`、`breakdown` 全 0

### 11.x 旧接口兼容映射（如后端已实现 `/carbon/footprint/*`）

> 若你们历史上已实现以下接口，建议“保留旧路径但内部转调/映射到新结构”，以避免前端改动与重复维护。

- `GET /carbon/footprint/` → 等价于 `GET /carbon/summary/?date=...`
- `GET /carbon/footprint/weekly/`、`GET /carbon/footprint/history/` → 轻量化阶段不做（如需可由 `CarbonSummary.week` 先承载）
- `GET /carbon/suggestions/` → 不做

## 14. 对后端的建议与注意事项

- **鉴权与错误码**
  - 建议所有需要登录的接口统一校验 `Authorization: Bearer {token}`，未登录或 token 失效时统一返回 `code = 401`。
  - 业务错误建议返回明确的 `message` 或 `msg` 字段，前端会直接展示给用户。

- **响应一致性**
  - 建议所有接口统一外层结构：`{ code, data, message }`。
  - `data` 字段存在时，前端会优先使用 `data`；否则退回使用整个响应对象。

- **可扩展性**
  - 目前大多数接口的 `data` 结构前端是宽松处理的，后端可以在不破坏已有字段的前提下适度扩展。

### 15.2 用户勋章关系表：`user_achievements`

- **用途**：记录用户已解锁的勋章，用于：
  - 生成个人荣誉墙 `GET /achievements/` 时的 `unlocked` / `unlocked_at`
  - 统计用户总积分、勋章数量等
- **建议字段**：
  - `id`（PK）
  - `user_id`（FK → 用户表）
  - `achievement_id`（FK → `achievements.id`）
  - `unlocked_at`（解锁时间）
  - `source`（可选：解锁来源，如 `challenge`, `report`, `carbon` 等）
  - 唯一约束：`(user_id, achievement_id)` 保证一人一勋章只记录一次

### 15.3 用户代表徽章表：`user_featured_badges`

- **用途**：存储用户「个性名片」中选择的代表徽章，对应前端的 `featured_badges`。
- **建议字段**：
  - `id`（PK）
  - `user_id`（FK → 用户表）
  - `achievement_id`（FK → `achievements.id`）
  - `sort_order`（排序序号，前端会按从小到大展示，最多 3 条）
  - 唯一约束：`(user_id, achievement_id)`，并建议对 `user_id, sort_order` 建立索引
- **前端约定**：
  - `GET /diet/profile/` / `GET /user/profile/` 中的 `featured_badges` 字段，即从该表关联 `achievements` 后生成 `BadgeSummary[]`。
  - 若用户未主动选择，后端可不返回该字段，前端会用 `badges` 的前 3 个补齐。

### 15.4 社区动态表（节选）：`community_feed`

- **用途**：为 `GET /community/feed/` 提供数据。
- **关键字段（与当前前端一致）**：
  - `id`（PK）
  - `user_id`（FK → 用户表）
  - `content`（文案）
  - `type`（`recipe` | `restaurant` | `meal` | `sport`，前端据此展示标签与关联卡片）
  - `image`（首图，列表/详情展示）
  - `sport_info`（当 `type = 'sport'` 时：`sport_type`, `type_name`, `duration?`, `distance?`, `calories?`）
  - `related_item`（当 type 为 recipe/restaurant 时的关联信息）
  - `is_saved`（当前用户是否已收藏该帖子）
  - `created_at`
- **与代表徽章的关系**：
  - `GET /community/feed/` 查询时，按 `community_feed.user_id` 关联：
    - `users`（拿到 `nickname`, `avatar` 等基础字段）
    - `user_featured_badges` + `achievements`（拼出 `featured_badges: BadgeSummary[]` 返回给前端）

### 15.5 小结

- 只要后端按以上结构或等价设计实现：
  - `GET /achievements/` 返回合并视图（成就定义 + 当前用户是否解锁）
  - `GET /diet/profile/` / `GET /user/profile/` 返回 `badges` / `featured_badges`（可选）及 `follow_count` / `fans_count` / `like_count`
  - `GET /community/feed/` 的 `user` 对象中带上 `featured_badges`（可选），动态含 `type`、`sport_info`（运动帖）、`is_saved` 等
  - 收藏相关：`GET /diet/favorites/`、`POST /diet/preference/` 与前端「我的收藏」、个人页收藏预览、菜谱/餐厅详情收藏联动
- 当前前端小程序即可完整打通：
  - 挑战完成 → 解锁勋章 → 荣誉墙展示 → 个人页个性名片 → 社区动态曝光；
  - 社区发帖（含运动记录）→ 动态流/帖子详情 → 用户主页 → 关注与粉丝；
  - 饮水记录（本地或可选后端同步）、健康管理日报与饮水展示联动。

## 17. 应季食材（文案与「自己做 / 冰箱」联动）

当前应季列表以本地 `detailMap` 为主；若后端提供应季接口，建议每条食材包含：

| 字段 | 说明 |
|------|------|
| `name` | 食材名 |
| `nutrition` | 营养价值长文案（可选 HTML 禁止，纯文本） |
| `season_why` | 「为什么当季」说明 |
| `recipe_ideas` | 推荐菜名数组（4–6 个），用于卡片芯片与跳转关键词 |
| `explain` | 一句话导语 |

前端行为：

- **加到冰箱**：`POST /diet/fridge/`（与现有冰箱一致）。
- **用它做菜 / 推荐做法芯片**：跳转 `packageCook/smart-choice/smart-choice`，Query：`fromSeasonalPage=true`、`ingredient=<食材名>`、可选 `dish_hint=<菜名关键词>`；智能决策页 **强制菜谱模式** `mode=cook`，并向 `POST /diet/search/` 传入 `filters.keyword`（优先 `dish_hint`，否则 `ingredient`）。后端可按关键词 + 食材表匹配菜谱。
- 用户先入库后，在「自己做」中切换 **匹配优先** 可结合 `fridge_ingredients` 提升命中（与现逻辑一致）。
