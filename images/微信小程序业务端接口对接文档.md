# 微信小程序业务端接口对接文档

更新时间：2026-04-10

适用对象：
- 微信小程序业务端前端
- 不适用于后台管理系统前端

后端业务接口根路径：

```text
http://127.0.0.1:8000
```

业务接口分两组：
- 用户与社交：`/api/v1/users/...`
- 饮食、社区、AI、报表等主业务：`/api/v1/diet/...`

说明：
- 后台管理接口 `/api/admin/...` 不在本文档范围内
- 旧的裸路径别名 `/api/v1/xxx` 已不再作为业务端标准路径
- 小程序前端统一按本文档中的完整路径对接

---

## 1. 深度分析结论

当前后端业务端架构可以分成两个核心域：

### 1.1 `users` 域

负责：
- 微信登录
- 登出
- 当前用户 Meta 信息
- 当前用户社交档案
- 指定用户主页
- 关注/取关
- 用户发帖列表

更适合用于：
- 登录态建立
- 我的页顶部统计
- 他人主页
- 社交关系

### 1.2 `diet` 域

负责：
- 健康档案
- 冰箱库存
- 搜索推荐
- 菜谱/餐厅详情
- 饮食日志 / 运动 / 体重 / 饮水
- 社区动态
- 挑战 / 成就 / 补救方案 / 碳足迹
- AI 能力
- 报表与图表

更适合用于：
- 首页
- 饮食记录
- 冰箱
- 社区
- 报表页
- AI 页

### 1.3 业务端当前最重要的接口规律

1. 绝大多数接口都使用统一响应结构：

```json
{
  "code": 200,
  "msg": "success",
  "data": {}
}
```

2. 全局默认需要 JWT 鉴权，只有登录接口匿名可访问。

3. 路由已经统一收口：
- 正确：`/api/v1/diet/achievements/`
- 错误：`/api/v1/achievements/`

4. 业务端里存在两个“个人档案”入口：
- `/api/v1/users/profile/`
- `/api/v1/diet/profile/`

它们都操作 `Profile`，但用途不同，前端不要混着用。

5. 有少量历史兼容接口的字段命名不完全统一，文档里已逐项标明。

## 2. 鉴权与公共约定

## 2.1 登录方式

### 微信登录

- 方法：`POST`
- 路径：`/api/v1/users/login/`
- 是否鉴权：否

请求体：

```json
{
  "code": "微信登录 code"
}
```

成功响应：

```json
{
  "code": 200,
  "msg": "success",
  "data": {
    "access_token": "xxx",
    "refresh_token": "xxx",
    "is_new_user": false,
    "user_id": 1
  }
}
```

前端注意：
- 小程序登录后统一使用 `data.access_token`
- 后续请求头：

```http
Authorization: Bearer <access_token>
```

### 开发环境测试登录

如果后端开启了 `ALLOW_TEST_WECHAT_LOGIN=true`，则可用：

```json
{
  "code": "TEST_任意字符串"
}
```

说明：
- 仅开发/联调环境可用
- 生产环境不应依赖该能力

## 2.2 登出

- 方法：`POST`
- 路径：`/api/v1/users/logout/`

请求体：

```json
{
  "refresh_token": "xxx"
}
```

## 2.3 统一响应规则

绝大多数接口返回：

```json
{
  "code": 200,
  "msg": "success",
  "data": {}
}
```

前端统一判断：
- 先看 HTTP 状态码
- 再看 JSON 中的 `code`

特别注意：
- `AI 食物识别` 在失败时为了兼容小程序 `wx.request`，可能返回 HTTP `200`，但 JSON 中 `code=500`
- 因此小程序请求封装不能只看 `statusCode`

## 2.4 图片 URL 规则

部分上传接口返回的 `url` 可能是：
- 绝对地址：`http://127.0.0.1:8000/media/...`
- 相对地址：`/media/...`

前端建议统一做 URL 归一化：

```js
function normalizeUrl(url) {
  if (!url) return ''
  if (url.startsWith('http://') || url.startsWith('https://')) return url
  return `${BASE_ORIGIN}${url}`
}
```

## 4. 用户与社交接口

## 4.1 获取当前用户 Meta

- 方法：`GET`
- 路径：`/api/v1/users/meta/`

用途：
- 我的页顶部信息
- 徽章、关注数、点赞数

响应字段：
- `nickname`
- `avatar`
- `days_joined`
- `goal_type`
- `daily_limit`
- `water_goal_cups`
- `water_goal_ml`
- `follow_count`
- `fans_count`
- `like_count`
- `badges`
- `featured_badges`

示例：

```json
{
  "code": 200,
  "data": {
    "nickname": "小明",
    "avatar": "http://127.0.0.1:8000/media/avatars/1.png",
    "days_joined": 32,
    "goal_type": "减脂",
    "daily_limit": 1800,
    "water_goal_cups": 8,
    "water_goal_ml": 2000,
    "follow_count": 12,
    "fans_count": 45,
    "like_count": 108,
    "badges": [],
    "featured_badges": []
  }
}
```

## 4.2 当前用户社交档案

- 方法：`GET`
- 路径：`/api/v1/users/profile/`

用途：
- 个人主页
- 头像与签名展示
- 社交统计展示

字段比 `diet/profile` 更偏社交展示。

### 头像上传

- 方法：`POST`
- 路径：`/api/v1/users/profile/`
- Content-Type：`multipart/form-data`

表单字段：
- `avatar`: 文件

### 修改社交档案

- 方法：`PATCH`
- 路径：`/api/v1/users/profile/`
- Content-Type：`application/json`

可更新字段示例：

```json
{
  "nickname": "新昵称",
  "signature": "今天也要认真吃饭"
}
```

## 4.3 他人主页

- 方法：`GET`
- 路径：`/api/v1/users/{userId}/profile/`

响应字段：
- `id`
- `nickname`
- `avatar`
- `signature`
- `follow_count`
- `fans_count`
- `like_count`
- `is_followed`

## 4.4 关注 / 取关

- 关注
  - 方法：`POST`
  - 路径：`/api/v1/users/{userId}/follow/`
- 取消关注
  - 方法：`DELETE`
  - 路径：`/api/v1/users/{userId}/follow/`

## 4.5 用户动态列表

- 方法：`GET`
- 路径：`/api/v1/users/{userId}/posts/`

查询参数：
- `page`
- `size`

响应结构：

```json
{
  "code": 200,
  "msg": "success",
  "data": {
    "list": [],
    "total": 20,
    "page": 1,
    "size": 10,
    "has_next": true
  }
}
```

## 5. 健康档案与偏好接口

## 5.1 推荐做法：业务资料编辑统一用 `/api/v1/diet/profile/`

原因：
- 这个接口更偏健康档案编辑
- 包含身高、体重、目标、饮食偏好、过敏原等业务字段
- 小程序业务端应优先使用它

## 5.2 获取健康档案

- 方法：`GET`
- 路径：`/api/v1/diet/profile/`

主要字段：
- `nickname`
- `avatar`
- `signature`
- `gender`
- `height`
- `weight`
- `age`
- `activity_level`
- `diet_tags`
- `allergens`
- `daily_kcal_limit`
- `goal_type`
- `target_weight`
- `water_goal_cups`
- `water_goal_ml`
- `bmr`

## 5.3 更新健康档案

- 方法：`POST`
- 路径：`/api/v1/diet/profile/`
- 说明：该接口将 `POST` 视作部分更新

请求体示例：

```json
{
  "nickname": "小明",
  "gender": "male",
  "height": 175,
  "weight": 70,
  "age": 28,
  "activity_level": "medium",
  "goal_type": "lose",
  "target_weight": 65,
  "diet_tags": ["低脂", "高蛋白"],
  "allergens": ["花生"],
  "water_goal_cups": 8
}
```

也支持文件上传头像：
- `multipart/form-data`
- 字段：`avatar`

## 5.4 偏好操作

- 方法：`POST`
- 路径：`/api/v1/diet/preference/`

请求体字段：
- `item_id` 或 `target_id`
- `item_type` 或 `target_type`
- `action`

支持类型：
- `recipe`
- `restaurant`
- `feed`
- `remedy`

支持动作：
- `favorite`
- `unfavorite`
- `like`
- `unlike`
- `block`
- `unblock`

返回：

```json
{
  "code": 200,
  "msg": "操作成功",
  "data": {
    "status": true,
    "action": "favorite"
  }
}
```

说明：
- `status=true` 表示本次操作后为“已存在”
- `status=false` 表示本次操作后为“已取消”

## 5.5 收藏列表

- 方法：`GET`
- 路径：`/api/v1/diet/favorites/`

查询参数：
- `type=all|recipe|restaurant|feed|remedy`

返回结构是统一聚合格式：
- `id`
- `type`
- `name`
- `image`
- `calories`
- `rating`

示例：

```json
{
  "code": 200,
  "msg": "success",
  "data": [
    {
      "id": "67f1...",
      "type": "recipe",
      "name": "鸡胸沙拉",
      "image": "",
      "calories": 320,
      "rating": null
    }
  ]
}
```

## 6. 冰箱接口

- 方法：`GET`
- 路径：`/api/v1/diet/fridge/`

查询参数：
- `search`

返回：

```json
{
  "code": 200,
  "msg": "success",
```

单项字段：
- `id`
- `name`
- `category`
- `sub_category`
- `quantity`
- `amount`
- `unit`
- `days_stored`
- `created_at`
- `expiry_date`
- `is_expiring`
- `is_scrap`
- `freshness`

## 6.2 新增冰箱食材

- 方法：`POST`
- 路径：`/api/v1/diet/fridge/`

请求体示例：

```json
{
  "name": "西红柿",
  "category": "蔬菜",
  "sub_category": "瓜果类",
  "quantity": "3个",
  "amount": 3,
  "unit": "个",
  "expiry_date": "2026-04-15",
  "is_scrap": false
}
```

返回：

```json
{
  "code": 200,
  "msg": "success",
  "data": {
    "item": {}
  }
}
```

## 6.3 单项详情 / 更新 / 删除

- 路径：`/api/v1/diet/fridge/{id}/`
- 方法：
  - `GET`
  - `PUT/PATCH`
  - `DELETE`

## 6.4 批量同步冰箱

- 方法：`POST`
- 路径：`/api/v1/diet/fridge/sync/`

请求体：

```json
{
  "operation": "override",
  "items": [
    {
      "name": "鸡蛋",
      "amount": 6,
      "unit": "个"
    }
  ]
}
```

说明：
- `operation` 默认 `override`
- `items` 必须是数组

## 7. 搜索发现接口

## 7.1 智能推荐搜索

- 方法：`POST`
- 路径：`/api/v1/diet/search/`

### 做饭推荐模式

请求体：

```json
{
  "mode": "cook",
  "page": 1,
  "sort_by": "match_score",

    "calorie_max": 500,
    "tags": ["低脂"],
    "cleanup_mode": true,

```

返回：

```json
{
  "code": 200,
  "data": {
    "recommendations": [
      {
        "id": "67f1...",
        "name": "鸡胸沙拉",
        "match_score": 90,
        "missing_ingredients": ["黑胡椒"],
        "cooking_time": 15,
        "difficulty": "简单",

        "image": "",

          }
        ],
        "match_reason": "匹配度高",
```

### 附近餐厅推荐模式

请求体：

## 8. 饮食 / 运动 / 体重 / 饮水接口

## 8.1 饮食打卡

- 方法：`POST`
- 路径：`/api/v1/diet/log/`

请求体字段：
- `source_type`
  - `1` 自制/菜谱
  - `2` 外卖/餐厅
  - `3` 自定义录入
- `source_id`
- `portion`
- `deduct_fridge`
- `meal_type`
- `meal_time`
- `macros`
- `calories`

示例：

```json
{
  "source_type": 1,
  "source_id": "67f1...",
  "portion": 1,
  "deduct_fridge": true,
  "meal_type": "lunch",
  "meal_time": "12:30",
  "macros": {
    "carbohydrates": 30,
    "protein": 20,
    "fat": 10
  }
}
```

返回：
- `log_id`
- `today_total_calories`
- `remaining_calories`
- `daily_summary`

## 8.2 删除饮食记录

- 方法：`DELETE`
- 路径：`/api/v1/diet/log/{id}/`

注意：

## 14. 小程序前端最容易踩坑的点

## 14.1 不要再使用旧裸路径

错误示例：
- `/api/v1/achievements/`
- `/api/v1/workout/today-stats/`
- `/api/v1/carbon/footprint/`

正确示例：
- `/api/v1/diet/achievements/`
- `/api/v1/diet/workout/today-stats/`
- `/api/v1/diet/carbon/footprint/`

## 14.2 `users/profile` 和 `diet/profile` 不要混用

推荐约定：
- 个人中心顶部信息：`/api/v1/users/meta/`
- 社交档案与头像：`/api/v1/users/profile/`
- 健康档案编辑：`/api/v1/diet/profile/`

## 14.3 上传接口的 URL 可能是相对路径

需要前端补全：
- `community/upload`
- `ai-nutritionist/upload`

## 14.4 AI 食物识别的错误不一定走 HTTP fail

前端必须判断：

```js
if (res.data.code !== 200) {
  // 当作错误处理
}
```

## 14.5 挑战任务列表和详情字段不一致

- 列表：`reward`
- 详情：`reward_points`

前端建议做兼容映射。

## 14.6 补救方案和成就部分字段有历史兼容痕迹

- 成就列表用 `name` / `description`
- 环保成就用 `title` / `desc`
- 补救方案列表用 `name` / `recipe`

建议前端加一层 normalize：

```js
const title = item.title || item.name || ''
const description = item.description || item.desc || item.recipe || ''
```

```js
function pickList(payload) {
  return payload?.data?.list
    || payload?.data?.items
    || payload?.data?.recommendations
    || payload?.data?.solutions
    || payload?.data
    || []
```

```js
function assertBizOk(res) {
  const code = res?.data?.code

  }
  return res.data
}
```
