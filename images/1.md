**四、401 与 404 的区别**
前端要明确区分，不要把两者都当成“接口坏了”。

1. `401 Unauthorized`
表示接口地址是对的，但当前请求没带有效登录态。
你日志里这两条就是正常现象：
- `/api/v1/diet/profile/`
- `/api/v1/diet/summary/`

因为随后登录成功后，它们都返回了 `200`。

前端处理建议：
- 登录前不要主动请求需要鉴权的页面数据
- 或者请求失败后检测 `401`，触发登录/刷新 token 流程

2. `404 Not Found`
表示前端请求的地址本身不对。
你日志里的这些就是典型地址错误：
- `/api/v1/workout/today-stats/`
- `/api/v1/achievements/`
- `/api/v1/ai-nutritionist/analyze/`
- `/api/v1/ai-nutritionist/warnings/`
- `/api/v1/carbon/footprint/`

前端处理建议：
- 先修 URL，不要把 404 当成后端鉴权问题

**五、登录返回结构已统一**
用户登录接口：

- 地址：`POST /api/v1/users/login/`

当前后端返回的核心字段是：

```json
{
  "code": 200,
  "msg": "success",
  "data": {
    "access_token": "...",
    "refresh_token": "...",
    "user_id": 1,
    "is_new_user": false
  }
}
```

前端注意：
- 不要再读取旧字段 `token`
- 应统一读取 `data.access_token`
- 鉴权请求头应使用：

```http
Authorization: Bearer <access_token>
```

**六、建议前端统一维护的接口前缀**
建议前端把接口按域拆成常量，避免后续再混。

示例：

```ts
const API_V1 = '/api/v1'
const USER_API = `${API_V1}/users`
const DIET_API = `${API_V1}/diet`
const ADMIN_API = '/api/admin'
```

然后按域拼接：

```ts
`${USER_API}/login/`
`${DIET_API}/profile/`
`${DIET_API}/workout/today-stats/`
`${DIET_API}/achievements/`
`${DIET_API}/carbon/footprint/`
```

不要再直接写这种裸地址：

```ts
'/api/v1/achievements/'
'/api/v1/workout/today-stats/'
'/api/v1/carbon/footprint/'
```
